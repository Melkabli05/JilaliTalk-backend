# Compatibility Considerations

Two independent contracts must survive this migration unchanged unless explicitly renegotiated: the **Angular-facing wire contract** (JSON shapes, HTTP methods/paths, WebSocket message shapes) and the **HelloTalk-upstream wire contract** (the request/response shapes HelloTalk's real backend expects and returns). The new domain model is free to be arbitrarily different internally — neither contract cares about `com.jilali.roomcontext`'s package structure — but every boundary-crossing byte must match.

## Angular frontend contract

- **Every HTTP method + path in `04-use-case-map.md` stays exactly as-is** during Phases 3-5 (a temporary `/v2` prefix or feature flag may be used for side-by-side verification per `07-migration-roadmap.md` Phase 4, but the final cutover in Phase 5 restores the original path). The Angular project (`JilaliTalk-angular-frontend`) should require zero code changes from this migration if this holds.
- **Response JSON shape must match field-for-field**, including the outbound camelCase conversion `CamelCaseResponseFilter` already performs (unchanged, this filter lives in `com.jilali.core` today and is explicitly NOT part of this migration's scope — it applies globally, to legacy and new endpoints alike, and stays where it is).
- **A `GET` endpoint that has a side effect stays a `GET`** (e.g. `join-bundle`) even though `04-use-case-map.md` classifies it internally as a Command — the use-case classification is an internal modeling decision, not a wire-contract change. Renegotiating HTTP semantics (e.g. making `join-bundle` a `POST` because "joining" is really a write) is explicitly out of scope; it would be a frontend-visible breaking change with no functional benefit, and the goal's own instruction is to preserve "existing functionality and API behavior."
- **WebSocket message shapes** (the `RoomRealtimeEvent`/`RoomCcRealtimeEvent` JSON the Angular frontend already parses) must be byte-identical even though internally they're now produced by `infrastructure.websocket.RoomEventTranslator` translating a `RoomEvent` domain event, rather than directly by the legacy `realtime.HtNotifyMapper`. The translation step is exactly where this compatibility guarantee is enforced — `RoomEventTranslator`'s outbound mapping must reproduce the legacy `HtNotifyMapper`'s exact output shape for every event type in `docs/audit/files/realtime/HtNotifyMapper.md`.

## HelloTalk upstream contract

- **Request shapes sent to HelloTalk must stay exactly what `infrastructure.client`'s wire DTOs already encode** — the new `infrastructure.mapper` layer's job is to translate `domain.model` ↔ these unchanged wire DTOs, not to redesign what's sent over the wire. The wire DTOs themselves (currently `room.dto.*`, `stage.dto.*`, etc.) become `infrastructure`-internal types in the new structure (see `08-class-mapping.md`) — their JSON shape (field names, `@JsonProperty` mappings, nullability) does not change, only their package location and their consumer (a mapper, not a controller directly).
- **Timestamp unit conventions must be preserved exactly**: HelloTalk sends Unix-seconds for most timestamp fields; the legacy `Comment.fromWireSeconds()` conversion (added during the prior refactor pass, `platform.time.Seconds.toMillis()`) is the authoritative reference for this conversion and must be reused, not reimplemented, in the new `infrastructure.mapper.CommentMapper`.
- **The `RewardItem` shape is frozen** at its current 8-field form (`platform.models.RewardItem`, already the canonical post-Refactor-3 shape) — reused directly by the new `domain.model.RoomLevel` value object, not redeclared.

## Backward compatibility during the coexistence window (Phases 2-5)

- Both the legacy `com.jilali.room`/`.stage`/etc. packages and the new `com.jilali.roomcontext` package will exist simultaneously in the same deployable artifact for the duration of Phases 2-5. This is safe because:
  - They are entirely separate Java packages with no shared mutable state (per `01-domain-model.md`, there's no local database either could contend over).
  - Both ultimately call the same HelloTalk upstream, which is the actual source of truth and arbiter of any conflict (see `09-technical-risks.md` R6 for the one real risk this coexistence introduces — simultaneous state-mutating calls from both paths against the same live session during verification).
  - Micronaut's DI container has no trouble hosting two independent bean graphs (`com.jilali.room.RoomController` and `com.jilali.roomcontext.api.RoomController` are different fully-qualified types, no naming collision at the container level) — though the HTTP route table WILL collide if both are mapped to the same path simultaneously, which is exactly why Phase 4's verification approach uses a temporary path prefix or feature-flag gate rather than registering both controllers on the same route.

## What is explicitly allowed to change (not a compatibility concern)

- Internal package structure, class names, method signatures **inside** `com.jilali.roomcontext` — none of this is observable from outside the JVM process.
- The `VipExperienceCard` state-machine tightening noted in `01-domain-model.md` (rejecting `use()` on an unclaimed/already-used card) is a **new validation that didn't exist in the legacy DTO-only model**. This is called out explicitly here because it's the one place this migration knowingly adds behavior the legacy code lacks. It is a business-rule correctness improvement (preventing a double-use bug class), not a security/authorization change, and does not require the durable "security is out of scope" instruction to be revisited — but it DOES mean a client that was previously able to call `use()` twice and get two (possibly inconsistent) responses will now get a rejection on the second call. If any legacy client behavior actually depends on that double-use tolerance (unlikely, but unverified), this would need to be revisited before Phase 5 cutover for the VIP capability.
