## Purpose

The JVM entry point for the whole `jilalibff` backend. Boots the Micronaut application context via `Micronaut.run(...)`.

## Responsibilities

- Start the Micronaut runtime, triggering classpath scanning, bean-definition loading (compile-time generated by Micronaut's annotation processor), and HTTP server startup.
- Nothing else. This class is intentionally inert.

## Public API

- `public static void main(String[] args)` — the JVM entry point, delegates immediately to `Micronaut.run(Application.class, args)`.
- Private no-arg constructor — blocks instantiation (this is a static-only launcher class, correctly marked `final` with a private constructor to prevent both subclassing and instantiation).

## Dependencies

- Depends on `io.micronaut.runtime.Micronaut` (framework bootstrap API) only.
- Everything else in the codebase depends on this indirectly (it's the root of the dependency graph in the sense that nothing runs without it), but no other class references `Application` directly — Micronaut's DI container is wired by annotation scanning, not by any class importing `Application`.

## Coupling and cohesion analysis

Maximally cohesive (does exactly one thing) and correctly decoupled — it has zero business-logic coupling to any feature package. This is the textbook-correct shape for a framework entry point.

## Code smells

None. This is an idiomatic, minimal Micronaut `Application` class.

## Technical debt

None found. Two minor observations, not debt:
- No `@OpenAPIDefinition` annotation is present here (a common pattern for centralizing Swagger/OpenAPI metadata on the Application class in Micronaut apps that use `micronaut-openapi`) — the codebase's `build.gradle`/`application.yml` does reference OpenAPI generation (per other agents' findings on `micronaut-openapi`), so if that metadata is declared elsewhere (e.g. a package-info.java), that's fine; if it's entirely absent, centralizing it here is a common, low-risk convention worth adopting.
- No explicit `@Factory`/`@Context` startup hooks live here, meaning eager-initialization concerns (if any) must be handled via `@Context`-scoped beans elsewhere in the codebase, not here — consistent with keeping this class inert.

None — single-purpose file, nothing to compare against.

None. `main` is the JVM entry point (invoked by the JVM launcher, not from application code — will show zero grep hits within `src/main/java` and that is expected/correct, not a dead-code finding).
